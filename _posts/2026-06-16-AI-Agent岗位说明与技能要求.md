# AI Agent 工程师 / AI 调试评测 / LLM Engineer 岗位说明与技能要求

**生成时间**: 2026-06-16  
**用途**: 记录当前招聘市场中 AI Agent 工程师、Agent 应用开发、AI 调试/评测/训练工程师、AI Coding / LLM Engineer 等岗位的职责、技能要求、学习方向与项目包装方式。

---

# 一、总体结论

现在招聘市场里，围绕大模型应用落地的岗位确实很热，但岗位名称非常混乱。不同公司可能叫：

- AI Agent 工程师
- AI Agent 开发工程师
- Agent 应用开发工程师
- AI 应用开发工程师
- 大模型应用工程师
- LLM Engineer
- AI Coding Engineer
- Agent Workflow 工程师
- AI 自动化工程师
- AI Agent 评测工程师
- AI Evaluation Engineer
- AI 调试工程师
- Agent Debugging Engineer
- AI 训练工程师
- AI Infra / 推理优化工程师
- AI 解决方案工程师

这些岗位不是完全一样，但核心都围绕一件事：

> **把大模型从“会聊天的模型”变成“能稳定完成任务的系统”。**

---

# 二、一句话解释这些岗位

## AI Agent 工程师

> 用大模型 + 工具 + 工作流 + 记忆 + 评测，把一个“会说话的模型”做成“能干活的自动化系统”。

比如用户说：

> 帮我整理客户退款申请，并处理符合条件的退款。

普通 ChatGPT 只能告诉你退款流程。  
AI Agent 系统要做到：

1. 理解用户意图
2. 查询订单 API
3. 查询客户历史记录
4. 判断是否符合退款规则
5. 调退款接口
6. 给客户发通知
7. 记录日志
8. 异常时转人工
9. 后续评估成功率和误判率

所以 AI Agent 工程师不是纯算法岗，也不是传统后端岗，而是：

> **软件工程 + LLM 应用 + 自动化流程 + 评测调试 + 安全控制** 的混合岗位。

# 三、主要岗位类型

## 1. AI Agent 应用开发工程师

这是目前最主流、门槛相对最低、需求最多的一类。

- 根据业务场景设计 Agent 流程

- 编写 Function Calling / Tool Use

- 做 RAG 知识库问答
- 用 Dify / Coze / LangChain / LangGraph / AutoGen / CrewAI 搭应用
- 调 Prompt
- 调试模型为什么乱调用工具、为什么答错、为什么死循环
- 上线后持续优化

### 典型业务场景

- 企业知识库问答
- 智能客服
- 自动写报告
- 自动处理工单
- 自动办公助手
- 销售线索分析
- 代码助手
- 数据分析助手
- 交易监控助手
- 运维自动化助手

### 常见 JD 关键词

- Python
- LLM
- Prompt Engineering
- RAG
- Function Calling
- MCP
- Workflow
- LangChain
- LangGraph
- Dify
- Coze
- AutoGen
- CrewAI
- API 集成
- Git
- 后端接口

### 适合人群

- 有 Python / Java / Go / Node.js 编程基础
- 做过后端、自动化、脚本、数据处理、爬虫、运维工具
- 能把业务流程拆成程序步骤
- 愿意做大量调试和集成工作

## 2. AI Agent 调试 / 评测工程师

这是新兴岗位，也非常有前途。

Agent 最大问题不是“能不能跑”，而是：

- 为什么乱调用工具？
- 为什么任务执行到一半跑偏？
- 为什么有时成功有时失败？
- 为什么同一个输入输出不稳定？
- 为什么上下文丢了？
- 为什么调用错 API？
- 为什么明明有工具却不用？
- 为什么成本突然飙升？
- 为什么出现幻觉？
- 为什么安全边界被绕过？

所以出现了专门的：

- Agent Evaluation Engineer
- AI 调试工程师
- AI 评测工程师
- LLM QA Engineer
- Agent Reliability Engineer

### 主要职责

- 设计测试集和评测集
- 分析 Agent 执行轨迹 trace
- 判断失败原因
- 做自动化评测
- 比较不同 Prompt / 模型 / 工具设计的效果
- 搭监控和告警
- 做灰度、A/B 测试
- 处理 hallucination、工具误用、权限越界、长上下文污染
- 做红队测试：诱导 Agent 犯错，看系统是否安全

### 和传统测试岗位的区别

传统软件测试是确定性的：

```text
输入 A → 输出 B
```

Agent 系统是非确定性的：

```text
输入 A → 可能调用搜索 → 可能查数据库 → 可能走 20 步 → 可能第 13 步跑偏
```

所以 AI 调试/评测工程师调试的不是普通代码，而是：

> **debug reasoning / debug trace / debug behavior**

### 核心能力

- 看懂 Agent trace
- 分析 tool call 是否合理
- 设计评测指标
- 构建 eval dataset
- 判断模型幻觉
- 判断 Prompt 是否失效
- 判断 RAG 是否检索错误
- 发现安全绕过路径
- 能把失败案例转化为回归测试

## 3. AI Agent 训练工程师 / AI 训练标注工程师

这个名字也很常见，但不同公司含义差别很大。

### 可能含义 A：偏数据训练 / 标注 / 对齐

主要做：

- 构造训练数据
- 清洗问答数据
- 标注模型输出好坏
- 写高质量示例
- 做 SFT / RLHF / DPO 数据准备
- 做模型评测
- 根据业务场景优化回答风格

这类岗位不一定要求很强代码能力，但要求：

- 表达能力强
- 逻辑清晰
- 会判断回答质量
- 对业务领域熟悉
- 有耐心做数据

### 可能含义 B：偏模型微调工程

主要做：

- LoRA / QLoRA 微调
- 数据集构建
- 模型训练脚本
- GPU 环境
- 模型评测
- 模型部署

技能要求更偏 ML：

- PyTorch
- Transformers
- datasets
- PEFT
- LoRA
- vLLM
- CUDA 基础
- GPU 调优

### 可能含义 C：偏 Agent 行为训练

主要做：

- 收集 Agent 失败案例
- 把失败案例做成训练/评测样本
- 调整 Prompt、工具描述、工作流
- 做 agent behavior tuning
- 设计安全和边界用例

这类和 Agent 评测工程师重叠度很高。

## 4. AI Coding Engineer / AI 编程工程师

这类岗位通常和“AI 写代码”“代码智能体”“软件工程自动化”相关。

### 主要职责

- 构建代码生成 / 代码修改 / 代码审查 Agent
- 接入代码仓库
- 自动读取项目上下文
- 自动运行测试
- 自动定位 bug
- 自动生成 patch
- 自动写 PR 描述
- 自动做 code review
- 构建 coding benchmark / eval

### 典型产品

- AI Coding Assistant
- Code Review Agent
- Bug Fix Agent
- Test Generation Agent
- DevOps Agent
- Repo Q&A Agent

### 需要技能

- Python / TypeScript / Go 等编程能力
- Git 原理
- AST / 代码解析基础
- 单元测试
- CI/CD
- sandbox 执行
- LLM tool use
- Agent trace
- patch 生成
- 编程任务评测

### 这个方向的难点

- 代码上下文很长
- 多文件修改容易漏
- 自动 patch 可能引入新 bug
- 需要运行测试验证
- 需要防止误删文件、泄露密钥、执行危险命令

## 5. LLM Engineer / 大模型应用工程师

这是一个大伞岗位，范围比 Agent 更广。

### 主要职责

- 调用大模型 API
- 构建大模型应用
- 设计 Prompt
- 做 RAG
- 做工具调用
- 做模型评测
- 做部署和监控
- 选择合适模型
- 控制 token 成本和延迟

### 和 AI Agent 工程师的关系

LLM Engineer 更宽，Agent Engineer 更偏“多步任务 + 工具执行 + 状态流”。

可以理解为：

```text
LLM Engineer ⊃ AI Agent Engineer
```

LLM Engineer 可能只做：

- 文案生成
- 问答
- 摘要
- 分类
- RAG

AI Agent Engineer 通常还要做：

- 规划
- 工具调用
- 多步执行
- 记忆
- 状态管理
- 人工确认
- 失败恢复

## 6. AI Agent 解决方案 / 交付工程师

这类偏业务和客户交付。

### 主要职责

- 用 Dify、Coze、百炼、扣子、FastGPT 等平台快速搭企业 Agent
- 和客户沟通需求
- 把客户流程拆成 Agent 工作流
- 接入客户系统
- 部署、培训、调优
- 写方案和文档
- 解决客户现场问题

### 技能要求

- 不一定算法强
- 但要会 Python / API / 数据库 / Docker / 基础后端
- 懂业务沟通
- 会快速落地 Demo
- 会排查系统问题
- 会写方案

### 适合人群

- 技术能做，沟通也不差
- 能快速理解业务
- 能把复杂系统讲清楚
- 愿意做项目交付

## 7. AI Infra / 推理优化工程师

这是高门槛底层岗，和 Agent 应用开发不是一回事。

### 主要职责

- 大模型推理加速
- vLLM / TensorRT-LLM / ONNX Runtime
- CUDA / C++ / 分布式推理
- GPU 性能优化
- KV cache 优化
- 高并发、低延迟、降成本
- 模型服务稳定性

### 适合人群

- C++ / CUDA / 系统底层强
- 做过高性能计算
- 做过模型部署
- 做过推荐/广告/搜索系统
- 算法和工程都很硬

### 特点

- 薪资高
- 门槛高
- 更偏底层工程
- 不适合作为零基础最快入门路线

# 五、这个岗位和传统程序员有什么不同？

传统程序员关注：

```text
需求 → 代码 → 测试 → 上线
```

AI Agent 工程师关注：

```text
业务目标 → 任务拆解 → 模型选择 → Prompt → 工具设计 → 记忆/RAG → 执行链路 → 评测 → 监控 → 迭代
```

最大的不同是：

> **Agent 系统不是完全确定性的。**

传统代码 bug 通常是：

- 空指针
- 类型错
- SQL 错
- 并发问题
- 超时

Agent 的 bug 更怪：

- 模型误解任务
- 工具描述写得不好
- 参数幻觉
- 记忆污染
- 检索结果不相关
- Prompt 太长导致重点丢失
- 多步任务中途偏航
- 调用了错误工具
- 明明失败却假装成功
- 某个模型升级后行为变了

所以好 Agent 工程师最值钱的能力是：

> **能把模糊、不稳定的大模型行为，工程化成稳定、可控、可观测、可评估的系统。**

# 六、AI 调试工程师具体调什么？

---

## 1. Prompt 问题

### 常见症状

- 输出格式不稳定
- 角色跑偏
- 工具调用太少或太多
- 不按业务规则
- 明明要求 JSON 却输出自然语言

### 常见解决办法

- 改系统提示词
- 加 few-shot
- 加 schema
- 加反例
- 拆 prompt
- 限制输出格式
- 做 Prompt 版本管理

---

## 2. 工具调用问题

### 常见症状

- 不该调用工具却调用了
- 应该调用工具却直接编答案
- 参数错
- 重复调用同一个工具
- 调用顺序不对
- 工具失败后没有补救

### 常见解决办法

- 重写工具描述
- 限制工具选择
- 加参数校验
- 工具返回结构化
- 增加 retry / fallback
- 拆分工具粒度
- 高风险工具加人工确认

---

## 3. RAG 问题

### 常见症状

- 查不到正确文档
- 检索到垃圾上下文
- 答案引用错
- 没有引用来源
- 模型基于错误上下文胡说

### 常见解决办法

- 调 chunk size
- 调 overlap
- 换 embedding 模型
- 加 rerank
- 混合检索：关键词 + 向量
- 做文档元数据过滤
- 建 RAG eval dataset

---

## 4. 多步任务问题

### 常见症状

- 执行到一半忘了目标
- 步骤重复
- 死循环
- 跳过关键步骤
- 中间结果污染最终结论
- 工具调用顺序混乱

### 常见解决办法

- 加状态机
- 加任务 checkpoint
- 限制最大步数
- 每步输出结构化
- 加自检节点
- 用 LangGraph 这种显式工作流
- 关键节点 human-in-the-loop

---

## 5. 评测问题

### 常见症状

- 不知道改动是变好还是变坏
- 靠人工感觉判断
- 上线后才发现坏了
- 改 Prompt 解决一个问题，又引入另一个问题

### 常见解决办法

建立 eval dataset，并设计指标：

- 任务成功率
- 工具调用准确率
- 幻觉率
- 格式合规率
- 平均成本
- 平均耗时
- 用户满意度
- 安全违规率

# 七、为什么这些岗位现在吃香？

## 1. 企业不缺模型，缺落地

现在 DeepSeek、通义、豆包、Claude、OpenAI 等模型都能用。  
企业真正缺的是：

> 怎么把模型接进业务系统里，还稳定可用？

这就是 Agent 工程师的活。

---

## 2. 低代码平台爆发，但复杂场景仍然需要工程师

Dify、Coze、百炼、FastGPT 等平台降低了门槛。  
但一旦业务复杂，就会遇到：

- 权限
- 数据源
- 异常处理
- 多系统集成
- 评测
- 成本
- 日志
- 部署
- 私有化
- 安全合规

这时不能只会拖拽，必须懂工程。

---

## 3. Agent 很容易 Demo 好看，上线难用

很多 Agent：

- 演示时很惊艳
- 真实用户一用就翻车

所以企业开始重视：

- Agent observability
- trace
- eval
- guardrails
- debugging

这就是 AI 调试 / 评测工程师吃香的原因。

---

## 4. 它比纯算法岗更容易转入

纯大模型训练要求：

- 深度学习
- 数学
- GPU
- 分布式训练
- 论文
- 模型结构

Agent 应用开发更看重：

- 编程
- 系统设计
- API
- 业务理解
- 快速落地
- 调试能力

所以对普通程序员更友好。

---

## 5. 有作品比学历更有说服力

很多 JD 都强调：

- 有 Demo 优先
- 有开源项目优先
- 有 AI Agent / RAG / 工具调用项目优先
- 有 GitHub 作品优先

这说明这个方向现在还处于“能做出来的人少”的阶段，作品很值钱。

# 十一、和当前能力/项目的匹配度

你最近做过的事情，其实已经很接近 Agent 工程的真实工作流：

| 现有实践 | 对应 Agent 岗能力 |
|---|---|
| OKX 策略脚本 | API 集成、自动化执行 |
| 海龟策略监控 | 状态管理、守护进程 |
| 日志排查 | observability |
| 下单权限检查 | 安全边界 |
| 实盘 / 模拟盘区分 | 环境隔离 |
| 持仓分析 | 数据获取 + LLM 分析 |
| 技能 / 脚本组织 | Tool / Skill 封装 |
| 修 stale PID | 生产稳定性 |
| 防止 `.env` 泄露 | 安全工程 |

所以如果未来包装简历，不要写：

> 我会用 AI。

而要写：

> 构建过基于 LLM + API + 自动化脚本的交易监控 / 策略执行 Agent，支持状态持久化、日志追踪、异常恢复、权限隔离、人工确认与实盘风控。

# 十二、岗位判断

这个方向是真的有前景，但要区分“真能力”和“伪风口”。

## 真需求

- 会 Python
- 会 API
- 会做业务系统
- 会接大模型
- 会 RAG
- 会工具调用
- 会调试和评测
- 能把 Agent 做稳定

## 伪需求

- 只会写 Prompt
- 只会用 Coze 拖几个节点
- 只会喊“智能体”
- 不懂代码
- 不懂后端
- 不懂测试
- 不懂部署
- 不懂安全

未来值钱的不是“会用 AI”，而是：

> **会把 AI 变成稳定生产力的人。**

# 十三、最终结论

AI Agent 工程师不是纯算法岗，而是未来的软件工程岗升级版。

最核心的竞争力是：

1. 会写代码
2. 会接系统
3. 懂大模型行为
4. 会调 Prompt 和工具调用
5. 会做 RAG
6. 会看 trace、做 eval
7. 会部署、监控、安全控制
8. 能把业务流程拆成 Agent 可执行步骤

最现实的成长路线是：

```text
Python 后端 / 自动化脚本
→ LLM API
→ RAG
→ Function Calling / MCP
→ LangGraph / Dify
→ Agent 调试评测
→ 生产级 Agent 项目
```

相比盲目卷大模型算法训练，这条路线更适合普通开发者，也更容易用项目证明能力。
